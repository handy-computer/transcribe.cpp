# Artifacts And Goldens

The repo policy is:

```text
Commit golden contracts, not golden payloads.
```

A committed contract says what should exist and how to reproduce it. The
binary payloads are generated or restored into the build tree or artifact
cache, then verified against the contract.

## Locations

Committed contracts:

```text
tests/golden/<family>/<case>.manifest.json
tests/tolerances/<family>.json
docs/porting/families/<family>.md
```

Generated or restored payloads:

```text
build/goldens/<family>/<case>/
build/validate/<family>/<case>/ref/
build/validate/<family>/<case>/cpp/
```

Reports:

```text
reports/validate/<family>/<case>.json
reports/reference/<family>/<case>.json
reports/bench/<machine>/<family>.<backend>.json
```

Optional local cache:

```text
$TRANSCRIBE_ARTIFACT_CACHE/
~/Library/Caches/transcribe.cpp/
~/.cache/transcribe.cpp/
```

The source tree should not rely on ignored `.f32`, `.bin`, `.npy`,
`.gguf`, or model files being present.

## Manifest

Each model variant should have a manifest like:

```json
{
  "schema": "transcribe-golden-manifest-v1",
  "family": "cohere",
  "variant": "cohere-transcribe-03-2026",
  "source_model": {
    "repo": "CohereLabs/cohere-transcribe-03-2026",
    "revision": "<hf-snapshot-revision>"
  },
  "canonical_reference": {
    "kind": "transformers-native",
    "path": "{path:transformers_ref}",
    "revision": "<transformers-git-sha>",
    "trust_remote_code": false
  },
  "paths": {
    "source_model": {
      "env": "TRANSCRIBE_COHERE_HF_MODEL",
      "default": "../models/cohere-transcribe-03-2026"
    }
  },
  "cases": [
    {
      "name": "jfk",
      "audio": "samples/jfk.wav",
      "stages": [
        {
          "name": "encoder",
          "out": "build/validate/cohere/cohere-transcribe-03-2026/jfk/encoder/ref",
          "command": ["uv", "run", "--project", "scripts/envs/cohere", "..."]
        }
      ]
    }
  ]
}

Stages are family-specific. Encoder-decoder models typically have
`mel`, `encoder`, and `decode` stages. Audio-LLM models (token
injection) may need an `inject` or `fuse` stage to validate the
audio-to-text boundary. CTC models have no decode stage. The
implementer defines stages during research based on the model's
architecture.
```

Generated payload directories receive a `golden-index.json` that lists
every tensor, shape, dtype, sha256, and non-tensor artifact for that run.
The manifest remains the committed contract; the index is generated next
to the payloads.

## Tensor Layout

Reference dumpers emit tensors in the canonical reference
implementation's natural row-major shape, with only the leading batch dim
squeezed for single-sample cases. They must not silently transpose or
reshape tensors to match a C++ implementation.

For a new family, generate these reference dumps before the C++ model
exists. The C++ debug dump path should then be implemented to match the
committed reference contract. If a layout adapter is unavoidable, make it
explicit in the dumper command or metadata and document it in the family
note.

Behavioral outputs such as transcripts are separate JSON artifacts, for
example `transcript.json`. They are generated by the reference path and
indexed next to the tensor payloads, but they are not tensor sidecars.

## Generation

The command shape is:

```bash
uv run scripts/golden.py generate \
  --manifest tests/golden/<family>/<variant>.manifest.json

uv run scripts/golden.py verify \
  --manifest tests/golden/<family>/<variant>.manifest.json
```

Current examples:

```bash
uv run scripts/golden.py plan \
  --manifest tests/golden/cohere/cohere-transcribe-03-2026.manifest.json

uv run scripts/golden.py generate \
  --manifest tests/golden/cohere/cohere-transcribe-03-2026.manifest.json

uv run scripts/dump_reference.py mel-onnx \
  --onnx <path-to-nemo128.onnx> \
  --audio samples/jfk.wav \
  --out build/goldens/parakeet/jfk-mel \
  --name jfk-mel

uv run scripts/dump_reference.py encoder \
  --model <path-to-parakeet-mlx-model> \
  --audio samples/jfk.wav \
  --out build/validate/parakeet/jfk/ref

uv run --project scripts/envs/cohere \
  scripts/dump_reference_cohere_transformers.py decode \
  --model <path-to-cohere-reference-model> \
  --transformers-ref <path-to-transformers-checkout> \
  --audio samples/jfk.wav \
  --out build/validate/cohere/jfk/ref \
  --language en \
  --max-new-tokens 256 \
  --model-dtype f32 \
  --torch-threads 1
```

## Default Tests

Default `ctest` must pass on a clean checkout without real model files.

If a test needs a generated golden, one of these must be true:

- CMake builds the golden first from repo-local or fetched inputs.
- The test skips with return code `77` when the manifest exists but the
  payload cannot be generated or restored.
- The payload is a deliberately committed tiny exception.

Do not register a test that unconditionally fails because an ignored
source-tree golden payload is absent.

## Cache Keys

Artifact cache keys should be derived from:

- family
- case
- manifest schema
- reference implementation and revision
- source model id and revision
- audio sha256
- dump-point list
- dtype and quantization policy

If the cache key or output sha256 does not match, regenerate or fail with
a clear command. Do not silently accept stale payloads.
